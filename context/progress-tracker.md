## Architecture Decisions

- `app/globals.css` defines two parallel token layers, both required: (1) shadcn's standard semantic vars (`--background`, `--foreground`, `--card`, `--primary`, `--secondary`, `--muted`, `--accent`, `--destructive`, `--border`, `--input`, `--ring`) so generated `components/ui/*` files render correctly untouched, and (2) the custom brand utilities from `ui-context.md` (`bg-base`, `bg-surface`, `text-copy-primary`, `border-surface-border`, `text-brand`, `bg-accent-dim`, etc.) for use in app-level components. Both layers point at the same underlying `--bg-*`/`--text-*`/`--accent-*` values, so they always stay in sync ŌĆö update the source variable, not each derived token, when the theme changes.
- This Next.js version renamed the `middleware.ts` file convention to `proxy.ts` (same behavior, new filename/export name). Route protection lives in `proxy.ts` at the project root via `@clerk/nextjs/server`'s `clerkMiddleware` + `createRouteMatcher`, not a `middleware.ts` file.
- Clerk's appearance API in the installed `@clerk/ui`/`@clerk/nextjs` versions uses `appearance.theme` (not the older `appearance.baseTheme`). All Clerk theme variables are defined once in `lib/clerk-appearance.ts` and referenced via `var(--token)` so they stay in sync with `globals.css` ŌĆö never hardcode a color for Clerk components.
- `bg-base` is defined via an explicit Tailwind v4 `@utility bg-base { background-color: var(--bg-base); }` in `globals.css`, not via `--color-base` in `@theme inline`. Registering a color under `@theme inline` auto-generates its *entire* utility family (`bg-`, `text-`, `border-`, `ring-`, etc.); naming one `base` collided with Tailwind's built-in `text-base` font-size utility (used by shadcn's `Input`/`Textarea`), silently forcing near-black text color on top of the correct font-size. If a future token needs the same treatment, prefer `@utility` for single-purpose color names that match a default Tailwind scale key (`base`, `sm`, `lg`, etc.) instead of registering them in `@theme inline`.
