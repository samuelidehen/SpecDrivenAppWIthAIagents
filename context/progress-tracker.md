## Architecture Decisions

- `app/globals.css` defines two parallel token layers, both required: (1) shadcn's standard semantic vars (`--background`, `--foreground`, `--card`, `--primary`, `--secondary`, `--muted`, `--accent`, `--destructive`, `--border`, `--input`, `--ring`) so generated `components/ui/*` files render correctly untouched, and (2) the custom brand utilities from `ui-context.md` (`bg-base`, `bg-surface`, `text-copy-primary`, `border-surface-border`, `text-brand`, `bg-accent-dim`, etc.) for use in app-level components. Both layers point at the same underlying `--bg-*`/`--text-*`/`--accent-*` values, so they always stay in sync — update the source variable, not each derived token, when the theme changes.
- This Next.js version renamed the `middleware.ts` file convention to `proxy.ts` (same behavior, new filename/export name). Route protection lives in `proxy.ts` at the project root via `@clerk/nextjs/server`'s `clerkMiddleware` + `createRouteMatcher`, not a `middleware.ts` file.
- Clerk's appearance API in the installed `@clerk/ui`/`@clerk/nextjs` versions uses `appearance.theme` (not the older `appearance.baseTheme`). All Clerk theme variables are defined once in `lib/clerk-appearance.ts` and referenced via `var(--token)` so they stay in sync with `globals.css` — never hardcode a color for Clerk components.
- `bg-base` is defined via an explicit Tailwind v4 `@utility bg-base { background-color: var(--bg-base); }` in `globals.css`, not via `--color-base` in `@theme inline`. Registering a color under `@theme inline` auto-generates its *entire* utility family (`bg-`, `text-`, `border-`, `ring-`, etc.); naming one `base` collided with Tailwind's built-in `text-base` font-size utility (used by shadcn's `Input`/`Textarea`), silently forcing near-black text color on top of the correct font-size. If a future token needs the same treatment, prefer `@utility` for single-purpose color names that match a default Tailwind scale key (`base`, `sm`, `lg`, etc.) instead of registering them in `@theme inline`.
- Prisma models live in `prisma/models/*.prisma` (multi-file schema), not appended to `prisma/schema.prisma`. `prisma.config.ts`'s `schema: "prisma/"` (directory, not a single file path) is what makes this work — it merges every `.prisma` file under `prisma/` at CLI-invocation time. `prisma/schema.prisma` keeps only the `generator`/`datasource` blocks; new models get their own file under `prisma/models/`, no generator block needed per file.
- `prisma.config.ts` loads env in two passes — `config()` for `.env`, then `config({ path: ".env.local", override: true })` — so the Prisma CLI's env precedence matches Next.js's own (`.env.local` wins over `.env`). Without the second call, the CLI silently falls back to whatever is in `.env` even when `.env.local` has the real value. If a third env file is ever added (e.g. `.env.production.local`), extend this chain rather than switching to a single bare `dotenv/config` call.
- `lib/prisma.ts` branches purely on the `DATABASE_URL` string prefix (`prisma+postgres://` → `accelerateUrl` constructor option; anything else → `@prisma/adapter-pg`'s `PrismaPg`), not on `NODE_ENV` or an explicit flag — so switching between a local Prisma Postgres dev server and a direct/pooled Postgres connection is just an env var change, no code change.
- `proxy.ts` no longer runs `auth.protect()` in front of `/api/*` (see the bug note under "Project API routes" above) — API routes are responsible for their own `await auth()` check and 401 response. `clerkMiddleware` still wraps every request and establishes the auth context regardless, so `auth()` inside a route handler works whether or not `.protect()` ran. Any new `app/api` route must do its own `auth()`/`userId` check; don't assume the proxy blocks unauthenticated API requests.
- `app/api/*/route.ts` handlers return JSON-only, consistent shapes: `{ project }` / `{ projects }` on success, `{ error }` on failure, using the `lib/api-response.ts` helpers for the shared 401/403/404/400 cases. Follow this shape for any new project-related route rather than inventing a new envelope.
- A `Project`'s `id` (Prisma `cuid()` by default) doubles as its Liveblocks room id: on create, the client generates `slugify(name)-<8-char-suffix>` and `POST /api/projects` accepts that as an optional `id` override (validated as slug-safe) instead of always letting Prisma default it. There is no separate `roomId` column — don't add one; pass an explicit `id` through instead, the same way `createProject` in `hooks/use-project-actions.ts` does.
- `app/editor/layout.tsx` and `app/editor/page.tsx` / `app/editor/[projectId]/page.tsx` share one layout, so any mutation that should update the sidebar (which the layout server-fetches and passes down as props) must call `router.refresh()` — a bare `router.push()` to a sibling route under the same layout can be served from Next's client Router Cache without re-running the layout's data fetch, leaving the sidebar stale. All three mutators in `hooks/use-project-actions.ts` do this now; keep doing it for any new mutation that touches project data.
- `lib/projects.ts` is the shared data-fetching module for project lists (`getOwnedProjects`, `getSharedProjects`) — used by both `app/api/projects/route.ts`'s `GET` and `app/editor/layout.tsx`'s server-side fetch. Add new list-shaped project queries here rather than inlining `prisma.project.findMany` calls at the call site.
- `lib/project-access.ts` is the shared module for Clerk-identity + project-access checks (`getCurrentIdentity`, `hasProjectAccess`) — used by `app/editor/[projectId]/page.tsx`. Any future route that needs "is this user the owner or a collaborator on this project" should call `hasProjectAccess` rather than re-deriving the owner/collaborator check inline (the existing inline version in that page, from before this unit, was folded into this helper).
- The workspace route's chrome is split across two navbars by design: the generic `EditorNavbar` (in `EditorShell`, shared by every `/editor/*` route — sidebar toggle + `UserButton`) stays project-agnostic, while project-specific chrome (project name, Share button, AI-sidebar toggle) lives in `WorkspaceNavbar`, rendered only inside `WorkspaceShell` by `app/editor/[projectId]/page.tsx`. Don't merge these — `EditorShell`/`EditorNavbar` render on `/editor` too, where there is no active project.
- `EditorShell` derives `activeProjectId` from `usePathname()` (matching `^\/editor\/([^/]+)$`) and passes it to `ProjectSidebar` → `ProjectListItem` as `isActive`, which is what highlights the current room in the sidebar. This is the only project-specific state `EditorShell` owns; it doesn't know the project's name or role beyond what's already in the `ownedProjects`/`sharedProjects` props passed down from the layout. Follow the same pattern (derive from `usePathname()`, don't thread a new prop through the layout) for any future per-route sidebar state.
- Collaborator routes split their auth check by verb, not by a single blanket rule: `GET /api/projects/[projectId]/collaborators` allows owner-or-collaborator via `hasProjectAccess` (read access), while `POST`/`DELETE` require `project.ownerId === identity.userId` directly (mutation requires ownership). Follow this per-verb split — not a single "is this user allowed to touch this project at all" check — for any future collaborator-adjacent route, since the spec draws the same read/write line.
- `lib/collaborators.ts`'s Clerk enrichment is read-only and best-effort: it never writes anything back to Clerk or to Postgres, and a `ProjectCollaborator` row is valid and functional (grants `hasProjectAccess`) whether or not a matching Clerk user is ever found. `getCollaborators` does one batched `getUserList({ emailAddress: [...] })` call for the whole list rather than one lookup per row — keep it batched if this function grows, since it's called on every dialog open and every invite/remove response.
- Liveblocks authenticates via **access tokens** (`liveblocks.prepareSession(userId, {...}).allow(room, session.FULL_ACCESS)`), not ID tokens (`identifyUser`) — this project's access control is fully resolved in Postgres (`hasProjectAccess`) before Liveblocks is consulted at all, so there's no Liveblocks-side group/org/user-access-list to maintain in parallel. `lib/liveblocks.ts`'s cached `Liveblocks` client and `app/api/liveblocks-auth/route.ts` are the only places that should construct a `Liveblocks` (node) client or issue a session — don't create a second one elsewhere.
- A project's Liveblocks room is created lazily, on first `POST /api/liveblocks-auth` for that project id, via `getOrCreateRoom(room, { defaultAccesses: [] })` — always private by default, since per-request access is granted entirely through the access-token's `session.allow(...)`, not room-level `defaultAccesses`/`usersAccesses`. Don't set non-empty `defaultAccesses` on a room; it would bypass the Postgres-backed access check.
- The collaborative canvas lives in two client components split by responsibility: `components/editor/canvas-room.tsx` owns Liveblocks room setup (`LiveblocksProvider`/`RoomProvider`/`ErrorBoundary`/`ClientSideSuspense`) and knows nothing about React Flow; `components/editor/canvas.tsx` owns the `useLiveblocksFlow` + `<ReactFlow>` wiring and knows nothing about how the room was set up (it just assumes it's rendered inside one). Keep this split for any future canvas work — don't merge room setup and flow rendering into one component.
- `types/canvas.ts` is the shared source for canvas-related types and canvas-wide constants (`NodeShape`, `CanvasNodeData`, `CanvasNode`, `CanvasEdge`, `DEFAULT_NODE_COLOR`). Any future node/edge work (custom rendering, a shape/color picker, AI-generated nodes) should import from here rather than re-declaring the shape union, node/edge generics, or default color inline.
- `lib/shapes.ts` is the shared source for shape *toolbar* data (`SHAPE_DEFINITIONS`, `SHAPE_DRAG_MIME_TYPE`, `ShapeDragPayload`) — separate from `types/canvas.ts` because it pairs each `NodeShape` with UI concerns (an icon, a default drag size) rather than pure data shape. Any future node-creation entry point (an AI-generated-node flow, a right-click "add shape" menu) should reuse `SHAPE_DEFINITIONS` for default sizes rather than re-deriving them.
- New canvas nodes are added by calling the Liveblocks-backed `onNodesChange` (from `useLiveblocksFlow`) with a standard React Flow `{ type: "add", item: node }` change — not a separate "add node" API. This is the same `OnNodesChange` handler already passed to `<ReactFlow onNodesChange>`, so any code that needs to programmatically add/remove/update nodes should go through this one function with the appropriate `NodeChange` object rather than trying to mutate Liveblocks Storage directly.
- `hooks/use-collaborators.ts` does not fetch-on-mount via `useEffect` — ESLint's `react-hooks/set-state-in-effect` rule flags calling `setState` synchronously inside an effect body (including the first line of an async function invoked from one). `loadCollaborators`/`resetForm` are instead called directly from the event handlers that open/close the dialog (`WorkspaceShell`'s `openShareDialog`/`closeShareDialog`), and `ShareDialog` itself stays presentational, taking all hook state as props. Prefer this event-driven-fetch pattern over an effect keyed on an `isOpen` boolean for any future "load data when a dialog/panel opens" case.
