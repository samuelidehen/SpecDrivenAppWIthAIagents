## Architecture Decisions

- `app/globals.css` defines two parallel token layers, both required: (1) shadcn's standard semantic vars (`--background`, `--foreground`, `--card`, `--primary`, `--secondary`, `--muted`, `--accent`, `--destructive`, `--border`, `--input`, `--ring`) so generated `components/ui/*` files render correctly untouched, and (2) the custom brand utilities from `ui-context.md` (`bg-base`, `bg-surface`, `text-copy-primary`, `border-surface-border`, `text-brand`, `bg-accent-dim`, etc.) for use in app-level components. Both layers point at the same underlying `--bg-*`/`--text-*`/`--accent-*` values, so they always stay in sync ‚Äî update the source variable, not each derived token, when the theme changes.
- This Next.js version renamed the `middleware.ts` file convention to `proxy.ts` (same behavior, new filename/export name). Route protection lives in `proxy.ts` at the project root via `@clerk/nextjs/server`'s `clerkMiddleware` + `createRouteMatcher`, not a `middleware.ts` file.
- Clerk's appearance API in the installed `@clerk/ui`/`@clerk/nextjs` versions uses `appearance.theme` (not the older `appearance.baseTheme`). All Clerk theme variables are defined once in `lib/clerk-appearance.ts` and referenced via `var(--token)` so they stay in sync with `globals.css` ‚Äî never hardcode a color for Clerk components.
- `bg-base` is defined via an explicit Tailwind v4 `@utility bg-base { background-color: var(--bg-base); }` in `globals.css`, not via `--color-base` in `@theme inline`. Registering a color under `@theme inline` auto-generates its *entire* utility family (`bg-`, `text-`, `border-`, `ring-`, etc.); naming one `base` collided with Tailwind's built-in `text-base` font-size utility (used by shadcn's `Input`/`Textarea`), silently forcing near-black text color on top of the correct font-size. If a future token needs the same treatment, prefer `@utility` for single-purpose color names that match a default Tailwind scale key (`base`, `sm`, `lg`, etc.) instead of registering them in `@theme inline`.
- Prisma models live in `prisma/models/*.prisma` (multi-file schema), not appended to `prisma/schema.prisma`. `prisma.config.ts`'s `schema: "prisma/"` (directory, not a single file path) is what makes this work ‚Äî it merges every `.prisma` file under `prisma/` at CLI-invocation time. `prisma/schema.prisma` keeps only the `generator`/`datasource` blocks; new models get their own file under `prisma/models/`, no generator block needed per file.
- `prisma.config.ts` loads env in two passes ‚Äî `config()` for `.env`, then `config({ path: ".env.local", override: true })` ‚Äî so the Prisma CLI's env precedence matches Next.js's own (`.env.local` wins over `.env`). Without the second call, the CLI silently falls back to whatever is in `.env` even when `.env.local` has the real value. If a third env file is ever added (e.g. `.env.production.local`), extend this chain rather than switching to a single bare `dotenv/config` call.
- `lib/prisma.ts` branches purely on the `DATABASE_URL` string prefix (`prisma+postgres://` ‚Üí `accelerateUrl` constructor option; anything else ‚Üí `@prisma/adapter-pg`'s `PrismaPg`), not on `NODE_ENV` or an explicit flag ‚Äî so switching between a local Prisma Postgres dev server and a direct/pooled Postgres connection is just an env var change, no code change.
- `proxy.ts` no longer runs `auth.protect()` in front of `/api/*` (see the bug note under "Project API routes" above) ‚Äî API routes are responsible for their own `await auth()` check and 401 response. `clerkMiddleware` still wraps every request and establishes the auth context regardless, so `auth()` inside a route handler works whether or not `.protect()` ran. Any new `app/api` route must do its own `auth()`/`userId` check; don't assume the proxy blocks unauthenticated API requests.
- `app/api/*/route.ts` handlers return JSON-only, consistent shapes: `{ project }` / `{ projects }` on success, `{ error }` on failure, using the `lib/api-response.ts` helpers for the shared 401/403/404/400 cases. Follow this shape for any new project-related route rather than inventing a new envelope.
- A `Project`'s `id` (Prisma `cuid()` by default) doubles as its Liveblocks room id: on create, the client generates `slugify(name)-<8-char-suffix>` and `POST /api/projects` accepts that as an optional `id` override (validated as slug-safe) instead of always letting Prisma default it. There is no separate `roomId` column ‚Äî don't add one; pass an explicit `id` through instead, the same way `createProject` in `hooks/use-project-actions.ts` does.
- `app/editor/layout.tsx` and `app/editor/page.tsx` / `app/editor/[projectId]/page.tsx` share one layout, so any mutation that should update the sidebar (which the layout server-fetches and passes down as props) must call `router.refresh()` ‚Äî a bare `router.push()` to a sibling route under the same layout can be served from Next's client Router Cache without re-running the layout's data fetch, leaving the sidebar stale. All three mutators in `hooks/use-project-actions.ts` do this now; keep doing it for any new mutation that touches project data.
- `lib/projects.ts` is the shared data-fetching module for project lists (`getOwnedProjects`, `getSharedProjects`) ‚Äî used by both `app/api/projects/route.ts`'s `GET` and `app/editor/layout.tsx`'s server-side fetch. Add new list-shaped project queries here rather than inlining `prisma.project.findMany` calls at the call site.
