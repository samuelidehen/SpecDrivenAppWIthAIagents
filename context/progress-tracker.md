## Architecture Decisions

- `app/globals.css` defines two parallel token layers, both required: (1) shadcn's standard semantic vars (`--background`, `--foreground`, `--card`, `--primary`, `--secondary`, `--muted`, `--accent`, `--destructive`, `--border`, `--input`, `--ring`) so generated `components/ui/*` files render correctly untouched, and (2) the custom brand utilities from `ui-context.md` (`bg-base`, `bg-surface`, `text-copy-primary`, `border-surface-border`, `text-brand`, `bg-accent-dim`, etc.) for use in app-level components. Both layers point at the same underlying `--bg-*`/`--text-*`/`--accent-*` values, so they always stay in sync — update the source variable, not each derived token, when the theme changes.
- This Next.js version renamed the `middleware.ts` file convention to `proxy.ts` (same behavior, new filename/export name). Route protection lives in `proxy.ts` at the project root via `@clerk/nextjs/server`'s `clerkMiddleware` + `createRouteMatcher`, not a `middleware.ts` file.
- Clerk's appearance API in the installed `@clerk/ui`/`@clerk/nextjs` versions uses `appearance.theme` (not the older `appearance.baseTheme`). All Clerk theme variables are defined once in `lib/clerk-appearance.ts` and referenced via `var(--token)` so they stay in sync with `globals.css` — never hardcode a color for Clerk components.
- `bg-base` is defined via an explicit Tailwind v4 `@utility bg-base { background-color: var(--bg-base); }` in `globals.css`, not via `--color-base` in `@theme inline`. Registering a color under `@theme inline` auto-generates its *entire* utility family (`bg-`, `text-`, `border-`, `ring-`, etc.); naming one `base` collided with Tailwind's built-in `text-base` font-size utility (used by shadcn's `Input`/`Textarea`), silently forcing near-black text color on top of the correct font-size. If a future token needs the same treatment, prefer `@utility` for single-purpose color names that match a default Tailwind scale key (`base`, `sm`, `lg`, etc.) instead of registering them in `@theme inline`.
- Prisma models live in `prisma/models/*.prisma` (multi-file schema), not appended to `prisma/schema.prisma`. `prisma.config.ts`'s `schema: "prisma/"` (directory, not a single file path) is what makes this work — it merges every `.prisma` file under `prisma/` at CLI-invocation time. `prisma/schema.prisma` keeps only the `generator`/`datasource` blocks; new models get their own file under `prisma/models/`, no generator block needed per file.
- `prisma.config.ts` loads env in two passes — `config()` for `.env`, then `config({ path: ".env.local", override: true })` — so the Prisma CLI's env precedence matches Next.js's own (`.env.local` wins over `.env`). Without the second call, the CLI silently falls back to whatever is in `.env` even when `.env.local` has the real value. If a third env file is ever added (e.g. `.env.production.local`), extend this chain rather than switching to a single bare `dotenv/config` call.
- `lib/prisma.ts` branches purely on the `DATABASE_URL` string prefix (`prisma+postgres://` → `accelerateUrl` constructor option; anything else → `@prisma/adapter-pg`'s `PrismaPg`), not on `NODE_ENV` or an explicit flag — so switching between a local Prisma Postgres dev server and a direct/pooled Postgres connection is just an env var change, no code change.
- `proxy.ts` no longer runs `auth.protect()` in front of `/api/*` (see the bug note under "Project API routes" above) — API routes are responsible for their own `await auth()` check and 401 response. `clerkMiddleware` still wraps every request and establishes the auth context regardless, so `auth()` inside a route handler works whether or not `.protect()` ran. Any new `app/api` route must do its own `auth()`/`userId` check; don't assume the proxy blocks unauthenticated API requests.
- `app/api/*/route.ts` handlers return JSON-only, consistent shapes: `{ project }` / `{ projects }` on success, `{ error }` on failure, using the `lib/api-response.ts` helpers for the shared 401/403/404/400 cases. Follow this shape for any new project-related route rather than inventing a new envelope.
- A `Project`'s `id` (Prisma `cuid()` by default) doubles as its Liveblocks room id: on create, the client generates `slugify(name)-<8-char-suffix>` and `POST /api/projects` accepts that as an optional `id` override (validated as slug-safe) instead of always letting Prisma default it. There is no separate `roomId` column — don't add one; pass an explicit `id` through instead, the same way `createProject` in `hooks/use-project-actions.ts` does.
- `app/editor/layout.tsx` and `app/editor/page.tsx` / `app/editor/[projectId]/page.tsx` share one layout, so any mutation that should update the sidebar (which the layout server-fetches and passes down as props) must call `router.refresh()` — a bare `router.push()` to a sibling route under the same layout can be served from Next's client Router Cache without re-running the layout's data fetch, leaving the sidebar stale. All three mutators in `hooks/use-project-actions.ts` do this now; keep doing it for any new mutation that touches project data.
- `lib/projects.ts` is the shared data-fetching module for project lists (`getOwnedProjects`, `getSharedProjects`) — used by both `app/api/projects/route.ts`'s `GET` and `app/editor/layout.tsx`'s server-side fetch. Add new list-shaped project queries here rather than inlining `prisma.project.findMany` calls at the call site.
- `lib/project-access.ts` is the shared module for Clerk-identity + project-access checks (`getCurrentIdentity`, `hasProjectAccess`) — used by `app/editor/[projectId]/page.tsx`. Any future route that needs "is this user the owner or a collaborator on this project" should call `hasProjectAccess` rather than re-deriving the owner/collaborator check inline (the existing inline version in that page, from before this unit, was folded into this helper).
- The workspace route's chrome is split across two navbars by design: the generic `EditorNavbar` (in `EditorShell`, shared by every `/editor/*` route — sidebar toggle + `UserButton`) stays project-agnostic, while project-specific chrome (project name, Share button, AI-sidebar toggle) lives in `WorkspaceNavbar`, rendered only inside `WorkspaceShell` by `app/editor/[projectId]/page.tsx`. Don't merge these — `EditorShell`/`EditorNavbar` render on `/editor` too, where there is no active project.
- `EditorShell` derives `activeProjectId` from `usePathname()` (matching `^\/editor\/([^/]+)$`) and passes it to `ProjectSidebar` → `ProjectListItem` as `isActive`, which is what highlights the current room in the sidebar. This is the only project-specific state `EditorShell` owns; it doesn't know the project's name or role beyond what's already in the `ownedProjects`/`sharedProjects` props passed down from the layout. Follow the same pattern (derive from `usePathname()`, don't thread a new prop through the layout) for any future per-route sidebar state.
- Collaborator routes split their auth check by verb, not by a single blanket rule: `GET /api/projects/[projectId]/collaborators` allows owner-or-collaborator via `hasProjectAccess` (read access), while `POST`/`DELETE` require `project.ownerId === identity.userId` directly (mutation requires ownership). Follow this per-verb split — not a single "is this user allowed to touch this project at all" check — for any future collaborator-adjacent route, since the spec draws the same read/write line.
- `lib/collaborators.ts`'s Clerk enrichment is read-only and best-effort: it never writes anything back to Clerk or to Postgres, and a `ProjectCollaborator` row is valid and functional (grants `hasProjectAccess`) whether or not a matching Clerk user is ever found. `getCollaborators` does one batched `getUserList({ emailAddress: [...] })` call for the whole list rather than one lookup per row — keep it batched if this function grows, since it's called on every dialog open and every invite/remove response.
- Liveblocks authenticates via **access tokens** (`liveblocks.prepareSession(userId, {...}).allow(room, session.FULL_ACCESS)`), not ID tokens (`identifyUser`) — this project's access control is fully resolved in Postgres (`hasProjectAccess`) before Liveblocks is consulted at all, so there's no Liveblocks-side group/org/user-access-list to maintain in parallel. `lib/liveblocks.ts`'s cached `Liveblocks` client and `app/api/liveblocks-auth/route.ts` are the only places that should construct a `Liveblocks` (node) client or issue a session — don't create a second one elsewhere.
- A project's Liveblocks room is created lazily, on first `POST /api/liveblocks-auth` for that project id, via `getOrCreateRoom(room, { defaultAccesses: [] })` — always private by default, since per-request access is granted entirely through the access-token's `session.allow(...)`, not room-level `defaultAccesses`/`usersAccesses`. Don't set non-empty `defaultAccesses` on a room; it would bypass the Postgres-backed access check.
- The collaborative canvas lives in two client components split by responsibility: `components/editor/canvas-room.tsx` owns Liveblocks room setup (`LiveblocksProvider`/`RoomProvider`/`ErrorBoundary`/`ClientSideSuspense`) and knows nothing about React Flow; `components/editor/canvas.tsx` owns the `useLiveblocksFlow` + `<ReactFlow>` wiring and knows nothing about how the room was set up (it just assumes it's rendered inside one). Keep this split for any future canvas work — don't merge room setup and flow rendering into one component.
- `types/canvas.ts` is the shared source for canvas-related types and canvas-wide constants (`NodeShape`, `CanvasNodeData`, `CanvasNode`, `CanvasEdge`, `NODE_COLORS`, `DEFAULT_NODE_COLOR`, `DEFAULT_NODE_TEXT_COLOR`). Any future node/edge work (custom rendering, a shape/color picker, AI-generated nodes) should import from here rather than re-declaring the shape union, node/edge generics, or default colors inline. `NODE_COLORS` is the single source of the eight predefined background/text pairs from `ui-context.md` — a node's `color`/`textColor` should always be set from one of these pairs together, never independently, and `DEFAULT_NODE_COLOR`/`DEFAULT_NODE_TEXT_COLOR` are both derived from `NODE_COLORS[0]` rather than hardcoded separately.
- `lib/shapes.ts` is the shared source for shape *toolbar* data (`SHAPE_DEFINITIONS`, `SHAPE_DRAG_MIME_TYPE`, `ShapeDragPayload`) — separate from `types/canvas.ts` because it pairs each `NodeShape` with UI concerns (an icon, a default drag size) rather than pure data shape. Any future node-creation entry point (an AI-generated-node flow, a right-click "add shape" menu) should reuse `SHAPE_DEFINITIONS` for default sizes rather than re-deriving them.
- New canvas nodes are added by calling the Liveblocks-backed `onNodesChange` (from `useLiveblocksFlow`) with a standard React Flow `{ type: "add", item: node }` change — not a separate "add node" API. This is the same `OnNodesChange` handler already passed to `<ReactFlow onNodesChange>`, so any code that needs to programmatically add/remove/update nodes should go through this one function with the appropriate `NodeChange` object rather than trying to mutate Liveblocks Storage directly.
- `components/editor/shape-visual.tsx` (`ShapeVisual`) is the single source of shape-rendering logic (CSS shapes + the three SVG shapes), parameterized by `selected` for border/stroke color. Both `CanvasNodeRenderer` (the real node) and `ShapeDragPreview` (the drag-time ghost) render through it — any future change to how a shape looks (new shape type, different stroke treatment, etc.) belongs in this one file, not duplicated between the node renderer and the drag preview.
- The shape-panel's drag preview suppresses the browser's native drag ghost (`dataTransfer.setDragImage` with a 1×1 transparent GIF) and instead renders a React-portaled (`createPortal(..., document.body)`) `position: fixed` element tracked via the native `drag` event's `clientX`/`clientY`. This is the pattern to reuse for any future draggable-with-custom-preview UI in the app — don't rely on the default browser drag image if the dragged content needs to look like something other than a screenshot of the source element.
- `components/editor/canvas-node-context.tsx` (`CanvasNodeActionsProvider`/`useCanvasNodeActions`) is how node components reach back into the Liveblocks-backed canvas state owned by `CanvasFlow` — React Flow renders custom node types internally, so a node component can't be handed props directly by `CanvasFlow`; this context, provided just inside `<ReactFlow>` in `canvas.tsx`, is the sanctioned channel instead. Any future per-node action (delete, duplicate, color change, etc.) should be added to this same context's action bag rather than inventing a second context or prop-drilling through React Flow internals.
- All node-data mutations (label edits so far) go through `onNodesChange` with a `{ type: "replace", id, item }` change — the full node object, not a partial patch — built by spreading the current node found via `nodes.find(...)` and overwriting just the changed field in `data`. `NodeChange` has no partial "update data" variant, so any future per-node data edit (color, shape swap, etc.) should follow this same look-up-then-replace pattern in `CanvasFlow`, exposed through `CanvasNodeActionsProvider`.
- **`useLiveblocksFlow`'s `onNodesChange`/`onEdgesChange` silently ignore `{ type: "remove" }` changes** — confirmed by reading `@liveblocks/react-flow`'s actual source (`dist/lib/flow.js`), not assumed: their `applyNodeChanges`/`applyEdgeChanges` have a `case "remove": break;`, a deliberate no-op, unlike plain `@xyflow/react`'s `applyNodeChanges` where `"remove"` does delete. To remove nodes/edges from Liveblocks-backed canvas state, call the separate `onDelete({ nodes, edges })` mutation `useLiveblocksFlow` also returns (already wired to `<ReactFlow onDelete={onDelete}>` for the Delete-key path) — never build `"remove"` changes by hand. `components/editor/canvas.tsx`'s `importTemplate` is the reference implementation of a bulk clear (`onDelete({ nodes, edges })` for everything, then `"add"` changes for the replacement set).
- `<NodeResizer>` (from `@xyflow/react`) is rendered directly inside `CanvasNodeRenderer`, gated on `isVisible={selected}` — resize handles only ever appear for the selected node, never at rest. It requires no explicit wiring to `onNodesChange`; resize interactions flow through the same internal controlled-mode change pipeline as node dragging already did. Any future per-shape minimum size (e.g. a bigger minimum for cylinder's cap curve to stay legible) would be the one thing to special-case here — currently `minWidth`/`minHeight` are one global pair (60×40) for every shape.
- `<NodeToolbar>` (from `@xyflow/react`, used in `components/editor/node-color-toolbar.tsx`) is the pattern for any future "small floating control panel above a selected node" UI — it's already view-portaled and position-aware, so don't hand-roll positioning math for node-attached overlays the way `ShapeToolbar`'s bottom-of-canvas panel had to (that one isn't node-attached, so `NodeToolbar` didn't apply there).
- Inline label editing is a per-node local `isEditing` boolean (not lifted to `CanvasFlow` or Liveblocks Presence) — only one node can visibly be "in edit mode" from any given client's perspective at a time in practice (double-clicking a different node just opens a second independent `isEditing` state), and this is intentionally not synced as presence; if simultaneous multi-user "who's editing this label" indicators are ever wanted, that would need to move into Presence rather than component state.
- All 4 connection handles on `CanvasNodeRenderer` are `type="source"`, not a source/target pair per side — this only works because `connectionMode={ConnectionMode.Loose}` is set on `<ReactFlow>` (source-to-source connections are otherwise rejected in the default `Strict` mode). If connection mode is ever changed back to `Strict`, the handles would need to become source/target pairs per side instead, or connecting between two "source" handles would silently stop working.
- Edges render through the custom `edgeTypes.canvasEdge` (`components/editor/canvas-edge.tsx`), not a built-in xyflow edge type — `components/editor/canvas.tsx`'s `defaultEdgeOptions.type = "canvasEdge"` is what makes every new connection (from `useLiveblocksFlow`'s `onConnect`) use it without the creation code needing to set `type` itself. The connection-line *preview* (shown mid-drag, before an edge exists) is separately styled via `connectionLineType`/`connectionLineStyle` on `<ReactFlow>` since there's no edge object yet to attach a custom component to.
- `components/editor/canvas-edge-context.tsx` (`CanvasEdgeActionsProvider`/`useCanvasEdgeActions`) is the edge-side equivalent of `canvas-node-context.tsx` — same reasoning (custom edge components are rendered internally by React Flow, can't receive props directly from `CanvasFlow`). Edge-data mutations (label edits so far) follow the identical look-up-then-`{ type: "replace", id, item }` pattern used for nodes, via `onEdgesChange` instead of `onNodesChange`.
- `<BaseEdge>`'s `...props` passthrough (including event handlers like `onMouseEnter`/`onDoubleClick`) only reaches its *visible* path, never the wider invisible `interactionWidth` hit-area path it also renders — confirmed by reading the library source, not assumed. Any custom edge that wants hover/click to work across the full `interactionWidth` (not just the thin visible line) must attach those handlers to a wrapping element (e.g. a `<g>`) around `<BaseEdge>` instead of passing them as props to `<BaseEdge>` itself.
- Undo/redo is Liveblocks' own room history (`useUndo`/`useRedo`/`useCanUndo`/`useCanRedo` from `@liveblocks/react`), not a custom implementation — it automatically covers every mutation that goes through `onNodesChange`/`onEdgesChange`/`onDelete`/`onConnect` (create, move, resize, label edit, color change, connect, delete, template import) for free, since all four are `useMutation`-wrapped writes through the same Liveblocks Storage/history. Don't build a separate undo stack for canvas actions; any future mutation gets undo/redo automatically as long as it's dispatched through one of these four existing functions rather than touching Storage directly.
- `components/editor/starter-templates.ts` is the shared source for predefined canvas templates (`CanvasTemplate` type, `CANVAS_TEMPLATES` array) — built with local `templateNode`/`templateEdge` helpers that pull default sizes from `lib/shapes.ts` and colors from `types/canvas.ts`'s `NODE_COLORS`, so template node styling never drifts from what a real dropped/colored node looks like. `components/editor/template-preview.tsx` renders these templates at card-thumbnail scale by reusing `ShapeVisual` directly (not a separate lightweight shape renderer) — any future addition to `ShapeVisual`'s shape set automatically shows up correctly in template previews too.
- The starter-templates modal is a case study in "the trigger and the state live in different subtrees": the open button is in `WorkspaceNavbar` (outside the Liveblocks/`ReactFlowProvider` tree), but the import logic needs `nodes`/`edges`/`onNodesChange`/`onEdgesChange`/`onDelete`, which only exist inside `CanvasFlow`. Resolved by threading `isTemplatesModalOpen`/`onCloseTemplatesModal` down as plain props through `WorkspaceShell` → `CanvasRoom` → `Canvas` → `CanvasFlow` (mirroring how `projectId` already flows the same path) rather than introducing a new context. Follow this same prop-threading pattern for any future "navbar-triggered, canvas-state-consuming" feature — a context would be overkill for two props with one consumer.
- `hooks/use-keyboard-shortcuts.ts`'s `isEditableTarget` check (input/textarea/`isContentEditable`) is the one place that decides whether a global keyboard shortcut should fire — any future shortcut should be added inside this hook's `keydown` handler (after that guard), not as a second independent `window` listener elsewhere, so the editable-field exclusion stays centralized.
- The `Presence.thinking` field name (`liveblocks.config.ts`) is authoritative as of `19-presence-avatar-context.md` — it was `isThinking` from `10-liveblocks-setup.md` through `18-starter-template.md`. Nothing consumes it yet (no UI reads or writes `thinking`), so if a future unit adds an AI-thinking indicator, use `thinking`, not `isThinking`.
- Two separate `<UserButton>` instances now exist in the workspace view: one in `EditorNavbar` (shared across all `/editor/*` routes) and one in `components/editor/presence-bar.tsx` (canvas-view only, sized to match collaborator avatars via `appearance.elements.avatarBox`). Both point at the same signed-in Clerk session — this is intentional duplication for layout purposes, not a bug; don't try to unify them into one shared instance, since they live in structurally different parts of the tree (outside vs. inside the Liveblocks room) for good reason (see the next note).
- The presence bar and live cursors are structurally confined to the canvas view for free: both live inside `CanvasFlow`, which only renders inside the Liveblocks `RoomProvider` (mounted only by the workspace route, `app/editor/[projectId]/page.tsx` → `WorkspaceShell` → `CanvasRoom`). There was no need for an explicit "am I on the canvas route" check — anything that needs `useOthers()`/`useMyPresence()` simply can't render anywhere else. Follow this same "let the component tree do the scoping" approach for any future room-only UI rather than adding a route guard.
- `useOthers()` filtering by `other.id !== currentUserId` is required even though `useOthers()` already excludes the local connection — it does NOT exclude a *different browser tab/session* signed in as the same Clerk user, which shows up as a distinct "other" connection with a matching `id`. Any future feature that lists room participants by identity (not just by connection) should apply this same explicit id filter, not rely on `useOthers()` alone.
- Live cursors are positioned via `@xyflow/react`'s `<ViewportPortal>`, not manual `flowToScreenPosition` math — it renders children inside the same pannable/zoomable DOM node as nodes/edges, so a flow-space `transform: translate(x, y)` is correct for every viewer automatically. Cursor *position* is broadcast in flow-space (via `screenToFlowPosition` in `onPaneMouseMove`) for the same reason — screen-space coordinates would only line up for a viewer at the exact same pan/zoom as the broadcaster. Any future room-participant overlay tied to canvas coordinates (not screen coordinates) should use this same pair (`screenToFlowPosition` to write, `<ViewportPortal>` to render) rather than hand-rolling viewport-transform math.
- `hooks/use-collaborators.ts` does not fetch-on-mount via `useEffect` — ESLint's `react-hooks/set-state-in-effect` rule flags calling `setState` synchronously inside an effect body (including the first line of an async function invoked from one). `loadCollaborators`/`resetForm` are instead called directly from the event handlers that open/close the dialog (`WorkspaceShell`'s `openShareDialog`/`closeShareDialog`), and `ShareDialog` itself stays presentational, taking all hook state as props. Prefer this event-driven-fetch pattern over an effect keyed on an `isOpen` boolean for any future "load data when a dialog/panel opens" case.
